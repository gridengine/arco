Database Schema

This chapter contains information about the database schema of the @@ARCO_NAME@@ module

Tables

sge_job

The sge_job table contains one record per each new job, new array task, and parallel task started in a tightly integrated parallel job.

Non-array jobs will have the j_task_number set to -1. Each array task will result in one record for the parent shepherd job with j_task_number set to -1 and one record for each array task with the j_task_number set to > 0.

For versions prior to 6.0u7 each non-array job will have j_task_number set to -1 for pending job state and j_task_number set to 0 for all other states.

Each parallel task will have j_submission_time set to a default time 1969-12-31 19:00:00

A record is created as soon as a job, an array task, or a parallel task is scheduled.

A short description of @@GRID_ENGINE_NAME@@ jobs, array jobs, parallel jobs and their differences can be found in the Sun Grid Engine User's Guide. The Glossary may be especially useful as an introduction.

sge_job_usage

The sge_job_usage table holds the job's resource usage over time.

Normally, the job usage is recorded after a job has finished, but for long running jobs an intermediate reporting entry is created after a midnight. This results in multiple usage records for a particular job and allows for a fine-grained resource usage monitoring over time.

The intermediate records report the delta job resource usage for a period between start to end time (columns ju_start_time, ju_end_time).

The ju_exit_status is set to -1 if the job is still running, i.e. it is the intermediate record. For queries counting jobs, the condition "WHERE ju_exits_status != -1" needs to be added as not to include the intermediate records in the final query result.

sge_job_request

Stores resources that a job has requested.

Two types of requests (qsub options) are currently handled:

  1. -l resource requests, e.g. -l arch=solaris,mem_total=100M

    For each request one record is created.

  2. -q queue request, e.g. -q balrog.q

    One record is created containing “queue” as variable and the request contents as variable.

List of Derived Values

After some time period (e.g. one year), these values can even be further compressed to weekly or monthly values.
